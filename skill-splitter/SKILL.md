# Skill Splitter

1つのスキル(ルール集)を入力として受け取り、次の2つのスキルを出力する:

1. **generator スキル** — 生成時にコンテキストへ入れるルール。出力を作る側が従う。
2. **reviewer スキル** — 生成物を別呼び出しでレビューする側が使う、敵対的チェックリスト。

分割の狙いは、同一コンテキストでの自己レビューによる「自己正当化バイアス」を避け、レビュワーには最小限の情報(ルール+出力のみ)だけを渡せる構造にすること。

## ワークフロー

### Step 1: 入力スキルの棚卸し

入力スキルを読み、含まれる指示を1つずつ抽出してリスト化する。各指示を次の3種に分類する:

| 分類 | 判定基準 | 行き先 |
|---|---|---|
| A: 機械検証可能 | 正規表現・スキーマ・文字数などコードで決定的に判定できる | generator + reviewer の `scripts/`(LLM判定させない) |
| B: 二値判定可能 | LLMがYes/Noで安定して答えられる | generator + reviewer のチェックリスト |
| C: 曖昧 | 「丁寧に」「自然に」等、判定がブレる | generator のみ。reviewer に入れる場合はBに分解してから |

C分類の指示は、可能な限りB分類の複数項目に分解する。分解例:

- 「丁寧な文体で」→「敬体(です・ます)以外で終わる文が存在するか」「命令形の文が存在するか」
- 「簡潔に」→「N文字を超えているか」(→これはAに落ちる)「同じ内容を言い換えて繰り返している段落があるか」

分解しきれない残りは generator 側にのみ残し、reviewer のチェック対象から外す。理由をユーザーに報告すること。

### Step 2: generator スキルの生成

- 入力スキルの指示を**肯定形の行動指示**として整理する(「〜するな」より「〜する」を優先。ただし禁止事項はそのまま残してよい)。
- レビュー・自己チェックに関する記述は**すべて削除**する。生成側に「後でチェックされる」ことを意識させる文言も入れない(チェック回避の学習を誘発するため)。
- frontmatter の description には、元スキルのトリガー条件を引き継ぐ。

### Step 3: reviewer スキルの生成

reviewer スキルは以下の構造にする:

- **役割宣言**: 「あなたはレビュワーである。生成物とルールのみが与えられる。生成の経緯・意図は考慮しない」
- **敵対的フレーミング**: 「問題がないか確認せよ」ではなく「違反を探せ。違反が見つからないのは例外的な結果である」と指示する。
- **チェックリスト**: B分類の項目をYes/No質問として列挙。1項目1観点。
- **引用の強制**: 違反と判定した項目には、生成物からの該当箇所の逐語引用を必須にする。引用できない違反報告は無効とする。
- **出力スキーマ**: 機械処理できる固定フォーマットを指定する。例:

```json
{
  "violations": [
    {
      "check_id": "C-03",
      "quote": "該当箇所の逐語引用",
      "reason": "違反理由の1文説明"
    }
  ],
  "passed": ["C-01", "C-02", "C-04"]
}
```

- チェック項目が10を超える場合はグループ分けし、「グループごとに別呼び出しでレビューすることを推奨」と reviewer スキル冒頭に明記する(注意の希薄化対策)。

### Step 4: A分類の機械チェック化

A分類の項目は、両スキルの `scripts/check.py` として決定的な検証スクリプトを書く。スクリプトは:

- stdin または引数で生成物テキストを受け取る
- 違反を Step 3 と同じJSONスキーマで stdout に出す(LLMレビュー結果とマージ可能にするため)
- 終了コード 0 = 違反なし / 1 = 違反あり

reviewer スキルには「LLMチェックの前にまず scripts/check.py を実行せよ」と明記する。

### Step 5: 出力と検収

出力ディレクトリ構造:

```text
<元スキル名>-generator/
├── SKILL.md
└── scripts/check.py   (A分類があれば)
<元スキル名>-reviewer/
├── SKILL.md
└── scripts/check.py   (同一のものを同梱)
```

最後にユーザーへ次を報告する:

1. A/B/C の分類結果一覧(元の指示 → 行き先)
2. Cのまま分解できず reviewer から漏れた項目(=ハーネスで担保されない項目)
3. 修正ループを組む場合の推奨: レビュー→修正の反復は2〜3回で打ち切ること

## アンチパターン(やらないこと)

- generator と reviewer に同一の長文ルールをコピペするだけの分割。reviewer 側は必ずYes/No質問形式に変換する。
- 曖昧項目(C)をそのままチェックリストに入れる。判定がブレて偽陽性・偽陰性の温床になる。
- reviewer に生成時の会話履歴や意図説明を渡すよう指示する。渡すのはルールと生成物のみ。
- 機械で判定できる項目(文字数・禁止語・フォーマット)をLLMに判定させる。
- このハーネスをセキュリティ境界(プロンプトインジェクション対策等)として扱う記述を出力に含めること。確率的な品質ガードレールである旨を reviewer スキルに明記する。
